深入长连接-websocket

Heartape 2023-06-05 websocket

# 1.简介

websocket集实时性与性能于一身，在im领域有其一席之地。java目前一般是采用netty和spring来实现。

# 2.netty

如果要实现websocket，netty当然是最好的选择之一。性能强劲，没有多余的代理，代码设计简洁优雅都是他的优点。

github

代码没有什么难点，只不过在返回消息时需要包装成TextWebSocketFrame，不然会收不到消息。。。

# 3.spring

虽然netty很优秀，但是考虑到spring强大的生态，如果不是作为一家公司的核心产品的话，spring将是首选，因为不需要造轮子可以快速开发。

github

代码没有使用注解的方式，与netty的代码几乎一致，包括返回消息也需要包装一下。 spring也提供了java标准注解的实现 上一篇文章

# 4.问题

如果使用websocket协议搭建im平台的话，不得不考虑鉴权的问题。 网络上的说法无非是将token放到param或者Sec-WebSocket-Protocol中，因为协议中其他请求头都不可以修改。 放到param有安全性问题，而放到Sec-WebSocket-Protocol则与其设计含义完全相悖，是极其不规范的操作。 从这里也可以看出，websocket在设计上就不打算在建立连接时做复杂操作。

我也找了一些商业产品，文档内介绍的做法是将token放到消息体内，也算是当前最合理的方案。

# 5.stomp

# 1.简介

鉴于上面的问题，用stomp是更为适合的方案。 STOMP，Streaming Text Orientated Message Protocol，是流文本定向消息协议，是一种为MOM(Message Oriented Middleware，面向消息的中间件)设计的简单文本协议。 很多消息中间件支持该协议，可以提高应用的兼容性和扩展性（spring就可以代理到外部中间件）。

# 3.解析

从http报文Sec-WebSocket-Protocol: v10.stomp, v11.stomp来看，stomp是作为websocket的一个子协议工作的。 stomp规定了消息的格式，而websocket并没有规定。

握手完成后，stomp客户端提供了3个操作:

  • CONNECT
  • SUBSCRIBE
  • SEND

对应服务端2种响应:

  • CONNECTED
  • MESSAGE

# 4.鉴权

stomp允许在消息中添加请求头，但并非在握手阶段传递，这一点因为其作为websocket子协议，两者一致。握手完成后会在 CONNECT 消息添加 Authorization:Bearer xxx -> user-name:1111 在提供token并解析后，spring会自动将user-name赋值并返回，如果是jwt就是subject的值。同时stomp客户端建议使用这个值作为订阅id。
