# match_phrase

  • match是全文搜索，会将检索关键词分词，仅需匹配任意一个词项。
  • match_phrase是短语搜索，也会将检索关键词分词，但是匹配所有词项，并且顺序和位置必须一致。而term不会分词。
  • match_phrase_prefix是一种特殊的match_phrase查询，它会对最后一个词项在倒排索引列表中进行通配符搜索，即相当于在match_phrase方式后增加了通配符的匹配。此项对性能消耗较高。
GET /book/_search
{
    "query": {
        "match_phrase": {
            "title.ik_smart_analyzer": "百年孤独"
        }
    }
}

wildcard的存在解决了一些问题：由于分词器词库的原因，导致全文检索时某些文档无法被匹配到，这时利用wildcard可以提高召回率。 wildcard属于精准匹配检索的范畴，但它有一些问题:

  • 性能低下，尤其是以通配符开头时（类似mysql like）。
  • 输入较长的字符串时，可能导致cpu大量消耗、请求错误甚至宕机，原因是lucene为了优化建立了一个DFA，它可能很复杂。

总之wildcard不建议使用，就算使用也需要严格控制字符串长度和字段数量。

替代:

  • Ngram
  • 7.9以上的版本，使用wildcard数据类型。

# multi_match

https://www.elastic.co/guide/en/elasticsearch/reference/current/query-dsl-multi-match-query.html
GET /book/_search
{
  "query": {
    "multi_match": {
      "query" : "百年孤独",
      // ^3修改该字段权重
      "fields" : [ "title^3", "introduce" ],
      // 指定查询类型
      "type" : "best_fields",
      // 字段权重系数
      "tie_breaker" : 0.0
    }
  }
}

type

best_fields: 默认类型，它会为每个字段生成一个match查询，并取得分最高的那个作为最终得分。
most_fields: 它会为每个字段生成一个match查询，并将所有字段的得分相加作为最终得分。
cross_fields: 它会将多个字段当作一个大的字段来进行匹配查询，适合用于处理同义词或多语言情况。
phrase: 它会使用match_phrase查询来匹配短语或近似短语。
phrase_prefix: 它会使用match_phrase_prefix查询来匹配以给定词项开头的短语。

# operator

GET /book/_search
{
    "query": {
        "match": {
            "title": {
                // term -> value
                // "value": "百年孤独",
                // match -> query
                "query": "百年孤",
                // 默认为or,表示所有分词之间的匹配结果是or的关系；如果设置为and,必须同时匹配所有分词
                "operator": "or",
                // 最少需要匹配2条分词, 只有operator为or时才生效
                // 百年孤独、百年好合，为2时都可以搜出，为3时只能搜出百年孤独
                "minimum_should_match": 2,
                // 修改当前查询语句的权重
                "boost": 2
            }
        }
    }
}

# 2.nested

nested嵌套文档会将从表数据保存到主表中，这种连接查询的方式性能较强，结构清晰，可读性强。 但是不适合修改较多或从表与主表的从属关系不明确（从表可能属于多个主表）的场景。

# 创建索引(book数据保存到bookshelf的文档中)

PUT /bookshelf
{
    "mappings": {
        "dynamic": "strict",
        "properties": {
            "id": {
                "type": "keyword",
                "index": "false"
            },
            "no": {
                "type": "keyword"
            },
            "book": {
                // 关键字会将该字段视为从表数据，否则搜索时会视为一个普通字段
                "type": "nested",
                "properties": {
                    "id": {
                        "type": "keyword",
                        "index": "false"
                    },
                    "bookshelf_id": {
                        "type": "keyword",
                        "index": "false"
                    },
                    "title": {
                        // text插入分词，配合term搜索不分词，可以部分匹配，配合match搜索分词，可以全文搜索
                        "type": "text"
                    },
                    "author": {
                        // keyword插入不分词，配合term搜索不分词，可以完全匹配
                        "type": "keyword"
                    },
                    "price": {
                        "type": "integer"
                    },
                    "publish_date": {
                        "type": "date",
                        "format": "strict_date || epoch_millis"
                    }
                }
            }
        }
    }
}

# 3.join

针对nested某些情况下不够灵活且数据冗余的问题，join或许是不错的方案。 原理就是将多个索引的数据存到一个索引里，通过join字段来区分。 官方文档也提到join相比nested性能相差较多，并且数据可读性非常差，需要按实际情况取舍。

# 创建bookshelf索引（相当于同时也创建了book索引）

PUT /bookshelf
{
    "mappings": {
        "properties": {
            "id": {
                "type": "keyword",
                "index": "false"
            },
            "no": {
                "type": "keyword"
            },
            "master_slave": {
                "type": "join",
                "relations": { 
                    // 定义了文档内的一组可能的关系，每个关系是父名和子名，可以定义多级join
                    "master": "slave"
                }

            // 下面是从表数据
            "book_id": {
                "type": "keyword",
                "index": "false"
            },
            "bookshelf_id": {
                "type": "keyword",
                "index": "false"
            },
            "title": {
                "type": "text"
            },
            "author": {
                "type": "keyword"

            "price": {
                "type": "integer",
                "index": "false"

对上面分词信息进行总结:

  • doc_freq: 索引中存在目标单词的文档数
  • ttf: 目标单词在所有文档中出现的频率（tff >= doc_freq）
  • term_freq: 单词在当前文档中出现的频率
  • position: 单词在文档中的位置
  • start_offset: 开始位置
  • end_offset: 结束位置

# 2.Posting List（倒排表）

在创建索引时，es会为text类型字段创建倒排表，与传统意义上的索引不同，倒排表保存了term对应的文档id，比如:

"百年" -> [1, 5, 8]

而不是如通常的索引一样在索引里寻找结果，本质上是空间换时间，所以整个字段的倒排表一定会膨胀许多。 自然，如此庞大的数据不可能全部放到内存，就算放到硬盘也是非常夸张的（数据量越大，数据就越分散，硬盘寻址越慢），所以自然需要压缩数据。

# Frame Of Reference

主要步骤如下:

  • 将倒排表转化为差值数组: [1, 5, 8, 56, 216] -> [1, 4, 3, 48, 160]
  • 计算是否需要拆分数组: 因为数组拆分后，需要一个byte（内存最小单位）来存储标志位（每一个元素占用的bit数），所以拆分越细，需要的标志位越多。
  • 拆分: 3 -> [1, 4, 3] , 7 -> [48, 160]

lucene为了存储bit数据，建立了 byte[][] 用来保存压缩数据,二维数组的优势在于扩容内存消耗较小

源码: org.apache.lucene.util.ByteBlockPool

# Roaring Bitmaps

RBM的主要思路是：将32位无符号整数按照高16位分桶（类似前缀），即最多可能有2^16=65536个桶，也可称为container。

伪代码

// 无符号整型
int b = unsigned(a);
// 拆分
short start = b >> 16;
short end = b & ((1 << 16) - 1);
// ArrayContainer,基于数量动态扩容。
short[] container = new short[];
// BitmapContainer，长度固定，length = ((1 << 16) - 1) >> 6 -> 1023 + 1 ，大小固定memory = length << 3 >> 10 = 8 byte
long[] container = new long[1024];
// 添加到对应桶
container.add(end);
// start -> 桶
containers.put(start, container);

# 3.Term Dictionary

就算进行了数据压缩，数据量依旧很大。 es底层选择通过建立Term Dictionary，将单词排序，并通过二分查找提升效率。

term Posting List
aa [1, 2]
ab [1, 3]
ba [4, 5]
bb [2, 4]

# 4.Term Index

如果数据量非常大，term的数量也会很大，将Term Dictionary全部放到内存中不现实。 Term Index 通过建立FST(Finite State Transducer)，以前缀指向Term Dictionary，极大缩小了term的查找范围。 Term Index数据量相比Term Dictionary会小很多，将Term Index放进内存可以接受。
