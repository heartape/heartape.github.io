线程池

vuePress-theme-reco Heartape    2022 - 2023

线程池

Heartape 2023-03-27 thread

# 1.简介

之前遇到了一个问题，服务的内存会有堆内存占用持续升高的情况的情况，只有重启才能解决。 原本以为是有内存泄漏，但是看了一下，代码里并没有什么骚操作，直到我看到了线程池。 这个接口用来记录一些简单数据，并发量相对较高，但并不是非常重要的业务。 所以为了优化这个接口对线程资源的占用，单独搞了一个线程池，以单个线程去处理这个业务，这就导致了问题。

# 2.代码

我搞了一个简单的程序做了模拟

@RestController
@RequestMapping("/thread/pool")
public class ThreadController {

    private final ThreadPoolExecutor threadPoolExecutor = new ThreadPoolExecutor(
            1,
            1,
            1,
            TimeUnit.MINUTES,
            new LinkedBlockingQueue<>()
    );

    /**
     * 模拟并发与耗时操作
     */
    @PostMapping("/executor")
    public void executor(){
        for (int i = 0; i < 10000000; i++) {
            threadPoolExecutor.execute(() -> {
                try {
                    TimeUnit.SECONDS.sleep(2);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            });
        }
    }
}

# 3.思考

对于以上的问题，该如何解决？

  • 使用有界队列：限定队列长度，但是超出部分会被丢弃导致接口降级。
  • CallerRunsPolicy：在有界队列的基础上改变拒绝策略，任务不会被丢弃，但是会加大对线程资源的挤占。
  • mq：有解耦、数据持久化、扩展性强等大量优势，但需要独立维护中间件。

在这个案例中，看似稳定的线程池也成为了不稳定因素，甚至创建线程池的本意也是为了优化。 所以深刻理解线程池是非常必要的。

# 1.ThreadPoolExecutor

线程池的使用较为简单，主要难点在于结合业务和流量做出正确的配置，特别是在流量波动较大的情况。

大多情况下，我们会在以下场景使用线程池:

  • 需要多线程的计算能力
  • 批量处理
  • 异步处理，减少前台等待时间

我们在使用线程池时还可能遇到的问题:

  • 使用无界队列或者队列长度太长，导致maximumPoolSize不生效，一直使用corePoolSize，导致无法扛住高流量。因为只有线程池队列占满才会使用非核心线程。
  • 线程数设置较小，导致处理速度太慢，接口下游调用大量超时，或者拒绝加入队列抛出异常导致接口降级。
  • 不支持优雅关闭，当项目关闭时，大量正在运行的线程池任务被丢弃。
  • 不支持运行时监控，使用不当可能会导致内存泄漏等问题。

# 2.ForkJoinPool

ForkJoinPool适合计算密集的场景，本质上是分治的思想，通过多线程将任务拆解。 对于 ForkJoinPool 来说，任务提交有两种：

  • 直接提交的外部任务:external/submissions task
  • fork分割的子任务:Worker task

fork/join 框架为我们提供了三类任务实现：

  • RecursiveAction：可用于没有返回结果的任务。
  • RecursiveTask ：用于有返回结果的任务。
  • CountedCompleter： 在任务完成执行后，触发自定义的钩子函数。

ForkJoinPool 的每个工作线程都维护着一个双端队列（WorkQueue），新的任务（fork分割的子任务）会放入工作队列的队尾，并从队尾取出任务来执行。 任务窃取: 每个工作线程在处理自己的工作队列同时，会尝试从其他任务队列队首窃取一个任务。

@RestController
@RequestMapping("/thread/pool")
public class ThreadController {

    private final ForkJoinPool forkJoinPool = new ForkJoinPool();

    @PostMapping("/fork")
    public void fork(@RequestParam Long count, @RequestParam Integer group){
        long l1 = System.currentTimeMillis();
        SumForkJoinTask sumForkJoinTask = new SumForkJoinTask(0L, count, group);
        ForkJoinTask<Long> forkJoinTask = forkJoinPool.submit(sumForkJoinTask);
        try {
            while (!forkJoinTask.isDone()){
                try {
                    TimeUnit.MILLISECONDS.sleep(1);
                } catch (InterruptedException e) {
                    throw new RuntimeException(e);
                }
            }
            System.out.println(forkJoinTask.get());
        } catch (InterruptedException | ExecutionException e) {
            throw new RuntimeException(e);
        }
        long l2 = System.currentTimeMillis();
        System.out.println(l2 - l1);
    }

    @AllArgsConstructor
    public static class SumForkJoinTask extends RecursiveTask<Long> {
        private Long start;
        private Long end;
        private Integer group;

        @Override
        protected Long compute() {
            if (this.end - this.start <= group){
                long total = 0;
                for (long i = this.start; i <= this.end; i++) {
                    total += i;
                }
                return total;
            }
            SumForkJoinTask sumForkJoinTask1 = new SumForkJoinTask(this.start, ((this.start + this.end) >> 1), group);
            sumForkJoinTask1.fork();
            SumForkJoinTask sumForkJoinTask2 = new SumForkJoinTask(((this.start + this.end) >> 1) + 1, this.end, group);
            sumForkJoinTask2.fork();
            return sumForkJoinTask1.join() + sumForkJoinTask2.join();
        }
    }
}
