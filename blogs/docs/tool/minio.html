# 1.简介

之前一直想搭建一个私人图床，但是找了好久，一般都是基于php或nodejs的较为复杂，但当我第一眼看到minio，就决定是它了。 MinIO 是一款高性能、分布式的对象存储系统，专为云原生设计。使用也是非常的广泛。

# 2.搭建

为了方便，我直接使用了docker搭建了单机版。

// 9000为后台api端口，9090为browser端口
docker run -p 9000:9000 -p 9090:9090 \
--name minio \
-d --restart=always \
-v /opt/minio/data:/data \
-v /opt/minio/config.env:/etc/config.env \
-e "MINIO_CONFIG_ENV_FILE=/etc/config.env" \
minio/minio server \
/data --console-address ":9090" -address ":9000"

# 3.分享链接

按照以上的方法启动容器后，就可以使用了。但是在分享链接的时候，却发现链接上并不是我们的域名，而是内网ip。进入dockers打印日志可以发现

[root@23fsw3213ffsd4t5 ~]# docker logs minio
API: http://172.xx.x.x:9000  http://127.0.0.1:9000
Console: http://172.xx.x.x:9090 http://127.0.0.1:9090
1
2
3

可以发现并没有指向我们的域名。

# 在config.env增加以下配置
MINIO_SERVER_URL=https://server.minio.heartape.com
MINIO_BROWSER_REDIRECT_URL=https://minio.heartape.com

再重启后查看日志就发现已经正常：

[root@23fsw3213ffsd4t5 ~]# docker logs minio

Console: https://minio.heartape.com

但是问题又来了，登录提示

invalid login

麻了，翻了好久发现中国这届网友不行，去github官方仓库找到了答案，原来需要在nginx增加配置

# 这里只列出了官方方案上的所有配置，9000同理，顺带解决了文件过大无法上传的问题
server {
    listen       443 ssl;
    listen  [::]:443 ssl;
    server_name  minio.heartape.com;
    
    ignore_invalid_headers off;
    client_max_body_size 0;
    proxy_buffering off;

    location / {
        proxy_pass http://172.24.xx.xxx:9090/;
        
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Host $http_host;

        proxy_connect_timeout 300;
        # Default is HTTP/1, keepalive is only enabled in HTTP/1.1
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        chunked_transfer_encoding off;
    }
}

# 4. 语法

{

        {
            "Effect": "Allow",
            "Action": [
                "s3:GetBucketLocation",

# 5.小结

本身minio搭建还是非常简单的，无数据库的方式（php图床需要）使得配置简单且易于迁移，总之就是非常nice。

鉴于其云原生的特性，后续如果有时间和需求，折腾一下kubernetes搭建minio集群。
